문제 제기: CSV는 왜 부족한가

1, 2주차에서 우리는 모든 결과를 CSV로 저장했다. 어제 분석은 어디 갔을까?

  • 덮어쓰기 — 새로 저장하면 어제 값이 사라진다
  • 타입 손실 — 숫자가 문자로 읽힌다
  • 검색 불가 — 특정 종목·기간만 뽑기 어렵다
  • 동시성 위험 — 두 노트북이 같이 쓰면 깨진다

분석가의 진짜 자산은 데이터가 아니다.

"누적된 데이터가 자산이다. CSV는 자산을 못 쌓는다."

핵심 개념: 데이터베이스란

표(Table)들의 모음

여러 개의 엑셀 시트가 한 파일에 모인 것. 시트끼리 연결도 가능하다.

타입 강제

숫자 칸에 문자가 들어오면 거부한다. 데이터 품질을 강제로 보장.

빠른 조회

인덱스 덕분에 100만 행에서도 즉시 원하는 행을 찾는다.

동시 안전

여러 프로그램이 동시에 써도 망가지지 않게 트랜잭션이 보호한다.

SQLite: 서버 없는 단일 파일 DB

  • 설치 불필요 — Python 표준 라이브러리에 내장(sqlite3)
  • 파일 하나 — quant.db 한 파일이 곧 DB 전체
  • 이식성 최강 — 그 파일을 USB로 옮기면 끝
  • 실전급 — Firefox, iOS, Android가 내부적으로 사용

"학습용이지만 장난감이 아니다."

개인 퀀트 시스템이 다루는 수만 행 규모는 SQLite로 충분히 감당된다.

SQL 4대 동사 (CRUD)

CREATE

새 표(Table)를 만든다. 컬럼 이름과 타입을 정의.

INSERT

표에 새 행(Row)을 추가한다. 매일의 시세가 여기서 들어간다.

SELECT

원하는 행을 뽑아본다. 분석의 99%는 SELECT다.

UPDATE / DELETE

고치거나 지운다. 잘못 쓰면 위험. 기록 시스템에선 거의 쓰지 않는다.

데이터 타입: 칸에 무엇을 넣을 것인가

SQLite 타입 의미 예시 퀀트 사용처
TEXT문자열"005930", "삼성전자"ticker, name, sector
INTEGER정수78500, 1500000volume, quantity
REAL실수(부동소수점)0.0234, 142.85weight, log_return
DATE실제는 TEXT (ISO8601 권장)"2026-04-29"거래일자, 분석시각

* SQLite는 실제 DATE 타입이 없다. "YYYY-MM-DD" 문자열로 저장하면 정렬·비교가 사전식으로도 정상 동작한다.

1단계: DB 파일 열기

파일이 없으면 만들고, 있으면 그냥 연다. 초기화 비용이 없다.

import sqlite3 conn = sqlite3.connect("quant.db") # 없으면 자동 생성 cur = conn.cursor() # SQL을 실행할 손잡이

연결(connection)은 DB 파일과의 통신 채널, 커서(cursor)는 그 채널로 SQL을 실어 보내는 도구다.

2단계: 일별 시세 테이블 만들기

cur.execute(""" CREATE TABLE IF NOT EXISTS prices ( ticker TEXT NOT NULL, date TEXT NOT NULL, open REAL, high REAL, low REAL, close REAL, volume INTEGER, PRIMARY KEY (ticker, date) ) """)

복합 PRIMARY KEY (ticker, date) — "한 종목의 한 거래일은 단 한 행"이라는 규칙을 DB가 강제한다. 같은 날을 두 번 넣으면 거부된다.

3단계: 데이터 넣기

# 한 행 cur.execute( "INSERT INTO prices VALUES (?, ?, ?, ?, ?, ?, ?)", ("005930", "2026-04-29", 78000, 79100, 77800, 78500, 12_345_678) ) # 여러 행 — pykrx로 받은 1년치를 한 번에 rows = [(t, d, o, h, l, c, v) for ... in df.itertuples()] cur.executemany("INSERT INTO prices VALUES (?,?,?,?,?,?,?)", rows) conn.commit() # 디스크에 확정 — 이걸 안 부르면 사라진다

* ? 플레이스홀더는 SQL 인젝션을 막는 표준 방어선이다. 절대 f-string으로 값을 직접 끼워 넣지 않는다.

4단계: 데이터 꺼내기

cur.execute(""" SELECT date, close, volume FROM prices WHERE ticker = ? ORDER BY date DESC LIMIT 5 """, ("005930",)) for row in cur.fetchall(): print(row)

SELECT + WHERE + ORDER BY + LIMIT. 이 4개 키워드만으로 분석의 80%가 가능하다.

pandas ↔ SQLite: 두 줄로 끝나는 다리

DataFrame → DB

df.to_sql( "prices", conn, if_exists="append", index=False )

DB → DataFrame

df = pd.read_sql_query( "SELECT * FROM prices " "WHERE ticker = ?", conn, params=("005930",) )

"DB는 pandas의 영구 저장 백엔드다. 메모리에서 굴리던 표를 그대로 디스크에 박는다."

5단계: 매일 누적 저장 (UPSERT)

매일 cron으로 돌리는 적재 스크립트는 중복 거부가 아니라 덮어쓰기 허용이 편하다.

# 같은 (ticker, date)가 있으면 갈아끼움, 없으면 새로 추가 cur.executemany(""" INSERT OR REPLACE INTO prices (ticker, date, open, high, low, close, volume) VALUES (?, ?, ?, ?, ?, ?, ?) """, rows) conn.commit()

INSERT OR REPLACE — UNIQUE/PK 충돌 시 이전 행을 지우고 새 값으로 덮는다. 정정공시·수정주가 반영에 유용.

6단계: AI 비평을 위한 별도 테이블

주가와 AI 출력은 성격이 다른 데이터다. 한 표에 섞지 않는다.

PHASE D: TIME TRAVEL & DESIGN

쌓인 데이터로 무엇을 할 것인가

"기록은 시작이다. 진짜는 비교에서 나온다."

7단계: 시계열 조회

# 최근 한 달치만 SELECT date, close FROM prices WHERE ticker = '005930' AND date BETWEEN '2026-03-29' AND '2026-04-29' ORDER BY date ASC;

BETWEEN은 양 끝을 포함하는 범위 필터다. ISO 날짜 문자열이라 사전식 비교가 그대로 시간 비교가 된다.

* ORDER BY date ASC(오래된 → 최신) 정렬은 차트 그리기 전 거의 항상 필요하다.

8단계: 집계 — GROUP BY

# 종목별 거래일 수 + 평균 종가 + 최고가 SELECT ticker, COUNT(*) AS days, AVG(close) AS avg_close, MAX(high) AS peak FROM prices GROUP BY ticker ORDER BY peak DESC;

2주차의 df.groupby("sector")["weight"].sum() — SQL의 GROUP BY가 그 원형이다. 같은 패턴, 다른 문법.

9단계: 과거 vs 현재 비교 쿼리

# 30일 전 종가와 오늘 종가를 한 행에 — 자기 자신과 JOIN SELECT today.ticker, today.date AS today, today.close AS close_now, past.close AS close_30d, (today.close - past.close) * 1.0 / past.close AS ret_30d FROM prices today JOIN prices past ON past.ticker = today.ticker AND past.date = date(today.date, '-30 day');

self-join — 같은 표를 두 번 부르면 "어제의 나"와 "오늘의 나"가 한 행에 나온다. 시계열 비교의 정석.

10단계: 누적 데이터의 진짜 가치

한 시점 차트는 정보, 여러 시점 차트는 통찰이다.

  • 종가 시계열 → 추세
  • 일별 수익률 분포 → 변동성
  • 섹터 비중 변화 → 포트 회전
  • AI 스코어 변화 → 의견 일관성

DB → pandas → matplotlib/plotly. 이 파이프라인이 1주차 + 2주차를 시간축으로 묶는다.

11단계: AI 비평의 시간 추적

# 같은 종목·페르소나에 대한 AI 스코어 추이 SELECT date, score, substr(text, 1, 40) AS preview FROM ai_reviews WHERE ticker = '005930' AND persona = '보수적 투자자' ORDER BY date ASC;

AI는 같은 차트도 날마다 다른 답을 줄 수 있다. 그 흔들림 자체가 분석 대상이다.

"AI 의견이 일관되지 않다는 사실 → 모델·프롬프트의 한계를 수치로 본다."

12단계: 정규화 (1NF · 2NF · 3NF)

"외울 필요 없다. 한 문장만 기억한다 — 중복을 줄이고, 한 사실은 한 곳에만 둔다."

1NF — 원자성

한 셀에 한 값. "NVDA, AMD"처럼 콤마로 묶지 않는다.

2NF — 부분종속 제거

복합 PK의 일부에만 매달리는 컬럼은 따로 뺀다. name은 ticker에만 의존 → 분리.

3NF — 이행종속 제거

"섹터명 → 섹터설명"처럼 비키 칼럼끼리 매달리면 따로 뺀다.

* 분석용 단일 표(분석 마트)는 일부러 비정규화하기도 한다. 원칙은 "기록 테이블은 정규화, 조회용 표는 펼친다."

13단계: 스키마 — 3개 테이블로 분리

종목 마스터. 이름·섹터는 여기 한 곳에만.

매일 적재. 시계열의 본체.

AI 출력 로그. 변경 없이 append-only.

"tickers가 정체성, prices가 사실, ai_reviews가 의견."

14단계: INDEX — 빠른 조회의 비밀

CREATE INDEX idx_prices_ticker_date ON prices(ticker, date); CREATE INDEX idx_reviews_ticker_date ON ai_reviews(ticker, date);

책 뒷부분의 색인(index)과 같다. 본문을 다 읽지 않고 단어 위치를 바로 찾는 장치.

  • WHERE / JOIN / ORDER BY에 자주 쓰는 컬럼이 인덱스 후보
  • 단점: 디스크 공간 + INSERT 약간 느려짐. 그래도 조회가 100배 빠르다

15단계: JOIN — 흩어진 표 합치기

# 시세 + 종목명 + 최근 AI 스코어를 한 표로 SELECT t.name, p.date, p.close, r.score, r.persona FROM prices p JOIN tickers t USING(ticker) LEFT JOIN ai_reviews r ON r.ticker = p.ticker AND r.date = p.date WHERE p.date = '2026-04-29' ORDER BY r.score DESC NULLS LAST;

INNER JOIN은 양쪽에 다 있는 행만, LEFT JOIN은 왼쪽 기준으로 모두 + 매칭만 합쳐온다.

16단계: 트랜잭션 (ACID)

A — Atomic

전부 성공 아니면 전부 롤백. 중간에 끊겨 반쪽 상태로 남지 않는다.

C — Consistent

제약 조건(NOT NULL, UNIQUE)이 항상 유지된다.

I — Isolated

동시에 도는 작업끼리 서로의 중간 상태를 안 본다.

D — Durable

commit한 결과는 정전이 나도 디스크에 살아남는다.

with conn: # 블록 끝에서 자동 commit / 예외 시 rollback cur.executemany(...) cur.execute(...)

17단계: 에러 다루기

  • UNIQUE 위반 — 같은 (ticker, date)를 또 넣을 때. 의도한 거면 INSERT OR REPLACE로 바꾼다.
  • NOT NULL 위반 — 필수 컬럼이 비었다. 입력 데이터 검증을 INSERT 전에 한다.
  • 타입 강제 실패 — 숫자 칸에 "N/A". pykrx가 결측을 내는 날을 처리해 둔다.
try: cur.executemany("INSERT INTO prices VALUES (?,?,?,?,?,?,?)", rows) conn.commit() except sqlite3.IntegrityError as e: print("중복 또는 NULL 위반:", e) conn.rollback()

18단계: 백업과 마이그레이션

백업

  • SQLite는 파일 하나 → cp quant.db quant.bak.db 끝
  • 매일 백업을 Drive/Git에 올리면 분석 이력이 영구화

마이그레이션

  • 컬럼 추가: ALTER TABLE prices ADD COLUMN adj_close REAL
  • 이름 변경/삭제는 SQLite에서 까다로움 → 새 표 만들고 INSERT…SELECT

운영 원칙 — 스키마를 바꿀 때는 먼저 백업. 분석가의 자산은 데이터다.

핵심 메시지

  • 분석은 스냅샷이 아니라 시계열이다. DB가 시간을 만들어 준다.
  • 스키마는 분석의 뼈대다. tickers / prices / ai_reviews — 성격이 다르면 표를 나눈다.
  • SQL의 SELECT + WHERE + GROUP BY + JOIN이 분석의 80%다. 외우지 말고 익숙해진다.
  • DB는 기록 도구이자 대화 도구다 — 과거의 나에게 질문할 수 있다.
